# Layout-RAG 业务扩展架构指南

本文档旨在阐述 Layout-RAG 系统如何利用**面向对象设计（OO Design）**实现核心排版逻辑与具体电气业务的解耦，并详细说明在接入新业务时如何进行特征定义、模版迁移及算法适配。

---

## 1. 核心设计哲学：引擎与领域分离

系统架构采用典型的**策略模式（Strategy Pattern）**，将“如何排版”的通用流程与“不同柜子怎么排”的业务知识分离。

-   **核心引擎（Engine）**：负责 RAG 检索流、坐标碰撞计算、WebAssembly 渲染加速等通用逻辑。它并不关心是在排配电箱还是高压柜，它只操作抽象的“元器件”对象。
-   **业务领域（Domain）**：封装了特定行业的专家经验（Domain Knowledge）。通过继承统一的业务基类，不同业务以“插件”形式挂载到引擎上。

---

## 2. 业务扩展的三大支柱

### 2.1 基于多态的特征提取 (Feature Architecture)
不同电气业务的“相似性”定义截然不同。系统通过面向对象的覆盖机制实现差异化检索：

-   **通用统计特征**：所有业务共享基础特征（如：元件总数、面板面积、填充率等）。
-   **领域专有特征提取**：每个业务类内部实现特定的提取算法。
    -   *低压柜业务*：重点提取主回路电流、进线极数、操作方式等对布局起决定作用的电气属性。
    -   *配电箱业务*：侧重识别“双电源切换”、“零地排分布”等由于结构紧凑导致的特殊约束特征。
-   **动态权重分配**：业务类不仅定义特征，还定义该业务下特征的“重要程度系数”，从而影响 RAG 检索时的 Top-K 排序。

### 2.2 跨业务模版迁移机制 (Template Migration)
当新业务接入但缺乏历史数据（冷启动）时，系统支持基于“基类兼容”的模版迁移：

-   **结构对齐逻辑**：系统在底层将所有业务的 BOM 数据抽象为统一的“逻辑树”。即便 A 业务的模板迁移到 B 业务使用，引擎仍能根据元件的功能属性（如：从“断路器”类派生）寻找最匹配的坐标锚点。
-   **位置重映射策略**：迁移过程中，系统会自动平移和缩放模板坐标，以适应新业务不同的面板尺寸，同时保持原始模板中体现的“逻辑关联”（如：指示灯总是在按钮上方）。

### 2.3 差异化约束适配 (Constraint Adaptation)
不同设备的装配工艺和安全标准不同，扩展时只需在业务类中声明不同的约束参数：

-   **安全净距策略**：高压柜可能需要更大的爬电距离约束，而低压柜则追求更高的排布密度。
-   **排列惩罚因子**：可以通过调整垂直/水平方向的惩罚倍数，引导算法生成“横向成排”或“纵向分组”的布局风格，无需修改核心算法代码。

## 3. 业务接入操作手册

接入一个全新业务（例如：直流桩、中压环网柜）的典型步骤：

1.  **定义身份标识**：指定唯一的业务 ID，为该业务分配独立的模板库和向量特征存储空间。
2.  **构建特征 Schema**：根据该行业的专家建议，挑选出影响排版的关键参数（如：电压等级、防护等级）。
3.  **覆写提取逻辑**：编写该业务专属的特征捕获方法，将复杂的 BOM 嵌套结构转化为检索器可识别的向量信号。
4.  **注入视觉语义**：配置特定业务的配色方案（例如：工业柜体通常使用灰/蓝色系，而监控屏可能使用高对比度色系）。
5.  **初始化服务**：在系统入口处将新业务对象注入 `LayoutService`，引擎即刻获得处理该业务的能力。

---

## 4. 扩展优势总结

-   **高复用性**：核心的求解器和 RAG 框架在所有业务中 100% 复用。
-   **零干涉**：新增业务只需增加新的类文件，不影响现有生产业务的稳定性。
-   **自进化**：随着各业务模版库的充实，每个业务的 RAG 准确度会独立进化。
